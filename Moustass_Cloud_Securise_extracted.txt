===== PAGE 1 =====
Projet Final: Moustass CloudSecL’application Missié Moustass a d’abord été développée comme une application utilisée exclusivement en interne par les employés de Barbichetz Ltd, une entreprise opérant dans le secteur financier.Aujourd’hui, face à une croissance attendue du nombre d’utilisateurs, notamment répartis à l’échelle internationale, Barbichetz Ltd souhaite migrer cette application vers le Cloud afin de la rendre accessible, maintenable et extensible. 


===== PAGE 2 =====
Contexte professionnel L’évolution des besoins de l’entreprise impliquent un changement d’architecture:•L’application devra désormais reposer sur une architecture cloud moderne, scalable et résiliente, capable de supporter un grand nombre d’utilisateurs simultanés.•Elle devra être dockerisée pour faciliter son déploiement sur différents environnements.•Chaque composant (frontend, backend, base de données) pourra être isolé, sécurisé et mis à l’échelle individuellement, conformément aux bonnes pratiques DevOps.
Ce projet représente donc une refonte complète:• Avec une attention particulière portée à la sécurité des échanges, •A la portabilité de l’environnement, et à la qualité du code, •Tout en assurant une expérience utilisateur fluide et sécurisée.

===== PAGE 9 =====
Un service = uneresponsabilité•Chaque microservice doit avoir un rôle unique et bien défini.•Un service ne doit jamais faire plusieurs choses différentes.
Application dans Mousatass: Auth Service (Spring Boot) gère uniquement :•authentification•mot de passe•JWT•Master Key•clés utilisateurs
Audio Service (Laravel) gère uniquement :•enregistrements vocaux•CRUD•chiffrement AES•hash SHA-256
Billing Service (Laravel) gère uniquement :•factures•statut de paiement•documents•Mauvaise pratique•Un service qui gère à la fois login + audio + facturation
Bonne pratique•Chaque service a une mission claire
D. Samfat Projet Moustass CloudSec9

===== PAGE 10 =====
Communication via API REST•Les microservices communiquent via HTTP avec des API REST. •Ils ne partagent pas leur code interne.
Application dans Mousatass react appelle :•Auth Service → login, register•Audio Service → CRUD audio•Billing Service → factures
Chaque requête contient :•Authorization: Bearer JWT
Exemple :•POST /api/auth/login•GET /api/audio•GET /api/invoices•Avantages:•Indépendance des services•Facilité de test (Postman, Swagger)•Compatibilité avec web et mobile
D. Samfat Projet Moustass CloudSec10

===== PAGE 11 =====
Isolation des données•Chaque microservice doit gérer ses propres données.•Il ne doit pas modifier directement les données d’un autre service.Application dans Mousatass•Auth Service•Table users•password•keys•email_verified
Audio Service•Table audio_records•hash•encrypted_data
Billing Service•Table invoices•amount•status•Mauvaise pratique•Audio Service modifie la table users
Bonne pratique•Audio Service utilise seulement user_id•Auth Service reste responsable des utilisateurs
D. Samfat Projet Moustass CloudSec11

===== PAGE 12 =====
Pas de logique métier partagée•Chaque règle métier doit être implémentée dans un seul service.•On ne duplique pas la logique.
Application dans Mousatass•Auth Service•Validation mot de passe•JWT•authentification
Audio Service•Chiffrement AES•Hash SHA-256•gestion fichiers audio
Billing Service•Calcul facture•statut paiement
Mauvaise pratique•JWT généré dans plusieurs services•Validation mot de passe dans Laravel
Bonne pratique•Chaque service gère sa logique métier
