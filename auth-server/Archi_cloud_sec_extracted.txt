Architecture Modèle Vue Constructeur (MVC) Modèle de conception logicielle qui sépare une application en trois composants principaux :1.Modèle: Gère les données et la logique métier.2.Vue : Affiche l’interface utilisateur et les données.3.Contrôleur : Gère les interactions entre la Vue et le Modèle.Origine :•Concept introduit dans les années 1970 (Smalltalk).•Utilisé dans les applications web et logicielles modernes.Objectif :•Séparer les responsabilités (Single Responsibility de SOLID)•Améliorer la maintenabilité, la scalabilité et la réutilisabilité du code.D. Fonctionnement du MVCModèle (Model)•Contient les données et la logique métier.•Base de données, classes métiers, requêtes SQL.•Exemple : Une classe Utilisateur avec des méthodes pour récupérer/modifier un profil.Vue (View)•Présente l’interface utilisateur (UI).•Fichiers HTML, CSS, templates (ex: Twig, React, Vue.js).•Ne contient pas de logique métier, seulement l’affichage des données.Contrôleur (Controller)•Reçoit les actions de l’utilisateur et agit en conséquence.•Ex : Un clic sur un bouton envoie une requête au contrôleur.•Met à jour le Modèle et choisit la Vue à afficher.
Exemple Web:1.L’utilisateur effectue une action (ex: soumettre un formulaire).2.Le Contrôleur reçoit l’action et interagit avec le Modèle.3.Le Modèle récupère les données et les renvoie au Contrôleur.4.Le Contrôleur met à jour la Vue et affiche les données.D. Architecture Client-Serveur 2-tiers (Two Tier)L’application est divisée en deux couches principales :1.Client (Frontend) : L’interface utilisateur et une partie de la logique applicative.2.Serveur (Backend + Base de données) : Gère la logique métier et l’accès aux données.Exemple : Une application de gestion de stock où un logiciel installé sur le poste utilisateur interroge directement une base de données SQL.Avantages :•Facile à développer et à maintenir.•Temps de réponse rapide pour un petit nombre d’utilisateurs.•Inconvénients :•Manque de scalabilité (peu adapté aux grandes applications avec de nombreux utilisateurs).•Sécurité limitée car le client accède directement à la base de données.

Architecture Client-Serveur3-tiers (Three-Tier)On ajoute une couche intermédiaire pour séparer la logique métier de la base de données:1.Client (Frontend / Présentation) : Interface utilisateur (navigateur web, application mobile, etc.).2.Serveur applicatif (Backend / Logique métier) : Traite les requêtes, applique la logique métier et interagit avec la base de données.3.Base de données (Données / Persistance) : Stocke les informations de l’application.Exemple :Un site e-commerce avec une interface web (HTML/CSS/JS), un serveur applicatif en PHP ou Node.js et une base de données MySQL.Avantages :•Séparation claire des responsabilités, facilitant la maintenance et l’évolutivité.•Meilleure sécurité car la base de données n’est pas directement exposée aux clients.•Permet une montée en charge plus facile en distribuant les composants.

Architecture n-tiers (Multi-Tiers)C’ est une extension du modèle 3-tiers en ajoutant davantage de couches pour une meilleure modularité et scalabilité:•Tier de présentation : Interface utilisateur (Web, mobile, desktop).•Tier de logique métier : API ou services qui gèrent les règles métiers.•Tier de services : Services externes comme l’authentification, le cache, les notifications, etc.•Tier de données : Bases de données, stockage de fichiers, etc.Exemple :Une application cloud avec une interface web (React), une API REST en Node.js, des microservices en Python et une base de données PostgreSQL.Avantages :•Très scalable et adapté aux grandes applications (SaaS, microservices, etc.).•Montée en charge•Sécurité renforcée grâce à l’isolation des couches.•Maintenance et mises à jour facilitées en modifiant uniquement des parties spécifiques du système.
Inconvénients :•Complexité accrue (déploiement, orchestration des services, communication entre tiers).•Risque de latence accru dû aux nombreuses interactions entre les différentes couches.D. Samfat -2025/26 Bachelor CDWFS - Bloc 03 156

Le Middleware (médiateur inter-couches )Assure la communication entre les différentes couches d’une applicationSe situe :•Sous l’applicatif•Au-dessus du système d’exploitation•Entre des composants nécessitant d’échanger des donnéesRôle principal : •Faciliter la liaison entre les couches de manière transparente•Masque les infrastructures réelles (réseaux, emplacements physiques…)•Cache les technologies sous-jacentes (protocoles, bases de données, systèmes de communication…)•Simplifie l’interopérabilité et l’intégration des composants
Exemples de Middleware:•Communication : Apache Kafka, RabbitMQ, MQTT (messagerie et streaming de données)•Bases de données : ODBC, JDBC, Hibernate (connexion et gestion des données)•Intégration d’applications : ESB (MuleSoft, Apache Camel), IBM WebSphere MQ (échange de données entre systèmes)•Web et API : GraphQL, API Gateway (Kong, AWS), Reverse Proxy (Nginx, Traefik)•Sécurité et Cloud : OAuth 2.0, Kerberos, Azure Service Bus (authentification et communication sécurisée)
